Internal and External Errors

Consider a class, WordDefinitions, that represents a dictionary, containing words and definitions. You can obtain the definition of a word by calling lookupDefn(), but only after init() is called. The init() method should only be called once. It reads all words and their definitions from a file, which is formatted as follows:

  • Each line contains a word-definition pair (though blank lines are also allowed, and ignored).
  • The word comes first, then a space, then the rest of the line represents the definition.

FYI on Initialisation

There can be good reasons for having some sort of “init” method, rather than performing all initialisation in the constructor. Constructors should not contain error-prone algorithmic logic (and ideally no significant algorithmic logic at all). If a constructor fails, there are sometimes ways to get hold of the resulting half-initialised object. These objects may be useful to an attacker as a way to manipulate your app into harming the user.

The discussion on exactly how this can happen is quite complicated, but worth pursuing.

The code, without any error handling, is as follows:

public class WordDefinitions
{
    private String filename;
    private Map<String,String> defns = null;

    public Definitions(String filename) 
    {
        this.filename = filename;
    }

    public void init()
    {
        defns = new HashMap<>();
        try(BufferedReader reader = new BufferedReader(new FileReader(filename)))
        {
            String line = reader.readLine().strip();
            if(!line.isEmpty())
            {
                String[] parts = line.split(" ", 2);
                defns.put(parts[0], parts[1]);
            }
        }
    }

    public String lookupDefn(String word)
    {
        return defns.get(word);
    }
}

Parsing Errors and Logging

Obtain a copy of worksheet4_matheval_java.zip. This simple app parses and evaluates mathematical expressions, but lacks error handling. Your job is to implement that error handling, as well as some logging.

Design

The only two classes you really need to focus on are ExprParser and Main.

ExprParser parses a string containing a mathematical expression, and in doing so, builds a tree of objects to represent the expression.

These expressions can consist of real numbers, a variable “x”, binary operators “+”, “-”, “*” and “/” (evaluated as per order-of-operators conventions), the unary negation operator “-”, and sub-expressions enclosed in “(” and “)”.

The parsing algorithm is divided between several “parse...()” methods, which are collectively recursive. That is, parse() calls parseAdd(), which calls parseMul(), which calls parsePrimary(), which can call parseAdd() again, as well as itself.

Main just reads an expression from the console, invokes ExprParser to build the tree, then evaluates the expression for different values of x.

Convince yourself that the app works (for valid expressions) by running it. It should work for any valid expression, within the limits of its features. But now consider invalid expressions.

  1. Try the following invalid expression strings.

    • 2 x
    • (x + 1&
    • ((x + 1
    • x + 1))
    • x + y
    • 1.2.3
    • x +
    • The empty string. (Just press enter.)

    Some of these will produce unhandled exceptions and raw stack traces, and others will produce results based on a subset of the expression. Neither of these outcomes is desirable.

    In all the above cases, what we want is for the user to be told “nicely” that the expression is invalid. (As a bonus, we’d also like to tell the user why the expression is invalid, but first things first.)

  2. Insert some logging statements inside the ExprParser class. You’ll need to:

    • Create the logger field.
    • Add a series of logger.info(...) and/or logger.fine() calls, particularly at the start of each method, and in each case/default branch of each switch statement.
    • In each case, ensure you’re logging the available information (e.g., the “tokens” list).

    Note

    The “matheval” app is configured for logging, if (as usual) you run it with “gradlew run”. The file “build.gradle” specifies that the file “logging.properties” contains logging configuration, and the latter specifies what the log levels are, and how to output log messages.

    Be aware that, without this configuation, logging statements won’t necessarily do anything!

    Note

    You can directly print/log a List, but the result may look confusing at first. Say your list contains ["(", "x", "+", "1", ")"]. When you convert this to a string (including if you add it to a string), you’ll get "[(, x, +, 1, )]". This can look nonsensical at first, until you realise that the parentheses are just ordinary strings, part of the list.

  3. Handle the errors.

    1. Since this is about external errors, first define a new custom exception class. DO NOT reuse an existing exception type.

    2. Add an appropriate throws clause to the method signatures in ExprParser. That’s where you’re going to throw your new exception type from.

      Note

      This is Java-specific. For other languages, you wouldn’t need to do this.

    3. The hard part: determine what error checking is missing in ExprParser, and add it in.

      • Consider the exceptions thrown (including the exact line number within the app) when you supply invalid expressions.

      • Consider what the logging statements are telling you.

      • HINT 1: several of the checks are going to concern whether the token list (the List<String>) is empty or not.

      • HINT 2: you can handle all the abovementioned invalid cases with around ~11 extra lines of code in ExprParser itself (not counting brace-only lines).

      Warning

      Don’t try to do error checking before parsing. It needs to be embedded within the parsing algorithm.

      Trying to do it up-front will either (a) be cheating, because you won’t be able to do it properly for all cases, or (b) result in a huge amount of extra unnecessary code.

    4. Try to ensure that each throw statement adds a relatively meaningful exception message; e.g., "unexpected token 'y'", if the expression contains “y”.

    5. Ensure Main has an appropriate catch block to handle your custom exception type, by printing out the message.
