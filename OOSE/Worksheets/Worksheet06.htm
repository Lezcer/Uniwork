Discussion: Application of the Observer Pattern

The Observer Pattern can technically be applied virtually any time you call a method without expectation of a return result (i.e., a void method, which also doesn’t send values back to the caller in other ways).

Consider this code:

public class X
{
    public void method1()
    {
        Y.method2();
    }
}


public class Y
{
    public static void method2()
    {
        System.out.println("Hello world");
    }
}


  1. Regardless of whether we should use the Observer Pattern here, we definitely can do so if we so choose. What changes would need to be made?

  2. What do we get out of doing this?

    All the extra code and declarations are an obvious downside in an example this small. But what’s the upside?

  3. Is the Observer Pattern pointless if there’s only one observer?

Theoretical Design

Consider the following problem: a satellite-based weather monitoring system is being developed to detect (1) the formation and (2) the disappearance of tropical storms (including cyclones, hurricanes or typhoons) around the world. Much of the system already exists, but it’s suggested that the Observer Pattern be used to help connect different parts together.

To begin with, the system will have a Satellite class that communicates with weather satellites to perform the actual detection. Appearance and disappearance events both come with latitude and longitude (real numbers), to mark the storm’s location.

When a storm is detected, several things must happen, all of which use the location information:

  • The system must record the incident in a particular database table.
  • The system must email a notification to the local weather agency.
  • The system must alert the user, showing the storm on a map.

There is an existing class in the system responsible for each of these actions: WeatherDB, Emailer and UI.

When a storm _dis_appears, the system must record the incident (including location) in the database, but no other actions are required. However, other kinds of actions may eventually be required in the future, for both formation and disappearance events.

  1. Draw a UML class diagram to demonstrate how to apply the Observer Pattern to this problem.

  2. Why is the Observer Pattern is useful here? What would the system look like without the Observer Pattern, and why would that design be a poor choice?

  3. Write a few lines of code to demonstrate how to set up the relationships between the observer and event source objects.

Implementing the Observer Pattern

Obtain a copy of worksheet6_reminder_java.zip. This is a simple, but somewhat incomplete todo/reminder application. The application is designed as follows:

There is also a ReminderApp class, containing the main method.

In the design so far, the AddReminderWindow can tell the controller to add a reminder, MainWindow can tell the controller to remove a reminder, and the controller makes the changes to the model. What’s missing is this:

  • MainWindow does not show the updated reminder text when a change is made (though it has a JList remindersList object for displaying this). The user can technically add a reminder, but nothing will actually show up.

  • The file is not updated when a change is made, and we would like it to be auto-saved. At the moment, all changes will be lost when the program exits.

Your task is to use the Observer Pattern to fill in these gaps. For this purpose, you should treat the ReminderList class as the event source (subject). If you think about it, ReminderList knows when the above actions should happen; but actually doing them is outside its responsibility.

You will then need to consider the following:

  • What other functionality does ReminderList require?
  • Which classes are going to function as observers?
  • How (and where) are you going to initialise the links between event source and observers?
